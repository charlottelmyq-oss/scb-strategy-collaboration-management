# SCB 战略合作管理系统需求草案

## 1. 系统建设背景

Dowsure 与 SCB 的合作已从单一银行产品合作，扩展为覆盖账户基础设施、信贷产品、开户推荐、Digital Onboarding、财富管理、MFP、市场活动、资本合作等多条线的复合型战略合作。

当前协作方式主要依赖表格、线下会议、飞书文档及人工跟进，已难以支撑多项目、多客户、多指标、多团队的高效协同。

## 2. 系统建设目标

SCB 战略合作管理系统应服务于以下目标：

- 对 SCB 合作项目全生命周期进行数字化监控，包括目标、过程、财务、风险、复盘。
- 打通 CRM 客户池与银行合作流程，提升进件、开户、财富管理等业务转化效率。
- 以任务驱动跨部门协作，减少会议后遗漏和重复沟通。
- 沉淀 SCB 合作知识库，支持自动化梳理、检索、总结和汇报。
- 建立仪表盘，追踪业务数据、项目进度、风险预警和管理层关心的核心指标。
- 未来支持多资方数据隔离，为渣打、汇丰等不同资方的战略合作管理提供可复制底座。

## 2.1 项目定位

本系统的第一阶段定位为“战略资方管理 AI 系统的 SCB 样板间”。也就是说，SCB 是 0 到 1 的试点对象，系统建设目标是先围绕 SCB 的真实合作场景跑通 MVP，再逐步抽象为可复制的多资方管理底座。

系统本身不是 SCB 合作下的一条业务项目线，而是战略资方合作管理的基础设施。未来当系统对接汇丰银行、OCBC、PADB、Fusion Bank 等不同战略合作资方的知识库 / 数据库后，应能够基于各资方实际合作内容，智能生成匹配的项目线、管理看板、汇报视图和文档材料。

## 2.2 与 SCB 战略合作目标的关系

SCB 战略合作本身的 2026 年第三目标是“多产品协同收入闭环”，即围绕 eCAS、Dscale、开户、Digital Onboarding、Affluent、MFP 等合作方向，形成一个客户多产品转化体系。

本系统 MVP 则是支撑该目标落地的管理基础设施：通过知识库、客户进度、任务闭环、仪表盘和自动汇报，把多产品协同从人工经验推进，转化为可跟踪、可复盘、可规模化的管理机制。

## 3. 初步用户角色

| 角色 | 主要职责 | 权限范围 |
| --- | --- | --- |
| 系统管理员 | 配置资方、用户权限、系统运行规则 | 可跨资方 |
| PMO / 高管 | 总览项目进度、财务指标、风险预警、重点事项 | 本资方或管理权限范围内 |
| 业务负责人 / BD 负责人 | 管理客户池、项目推进、任务分配、业务转化 | 本资方 |
| 客户经理 / 销售 | 处理具体进件、开户、客户沟通、待办事项 | 本资方 |
| 风控 / 运营 | 监控在贷、逾期、流程卡点、责任分析与知识库更新 | 本资方 |

### 3.1 账号权限与飞书授权入口

系统最前端应先设计账号权限与资方工作区选择，再进入具体 PMO 工作台。

建议采用“飞书授权登录 + 资方工作区权限”的方式：

- 用户通过飞书 OAuth / 企业身份授权登录。
- 系统根据飞书用户身份、部门、角色或后台配置，判断用户可访问的资方工作区。
- 例如汇丰 PMO 登录后，只进入汇丰战略合作工作区；渣打 PMO 登录后，只进入 SCB 战略合作工作区。
- 同一套系统框架可以复用，但知识库、客户表、任务、聊天记录、会议纪要和汇报视图必须按资方隔离。
- 系统管理员可以配置资方、用户角色、飞书数据源、授权范围和跨资方汇总权限。

第一版建议支持以下账号类型：

| 账号类型 | 默认工作区 | 数据权限 |
| --- | --- | --- |
| 渣打 PMO | SCB 战略合作工作区 | 仅可查看 SCB 相关项目、文档、客户、任务、群聊记录和汇报 |
| 汇丰 PMO | HSBC 战略合作工作区 | 仅可查看 HSBC 相关项目、文档、客户、任务、群聊记录和汇报 |
| 管理层 / 合伙人 | 单资方或跨资方汇总视图 | 根据授权查看单一资方或多资方汇总指标 |
| 系统管理员 | 系统配置台 | 配置用户、资方、飞书数据源和权限规则，不默认进入业务详情 |

飞书授权后，系统可在用户授权范围内拉取并索引以下上下文：

- 飞书 Wiki / 云文档：战略背景、会议材料、产品方案、操作指南、PRD 和汇报材料。
- 飞书群聊记录：指定合作群、项目群、客户推进群中的关键承诺、卡点、待办和决策。
- 飞书任务：任务标题、负责人、截止时间、状态、关联项目和提醒记录。
- 飞书多维表格 Base：客户推进表、项目数据表、活动名单、物料版本表和风险台账。
- 飞书日历 / 会议纪要：会议安排、参会人、会议摘要和 Follow-up。

权限原则：

- AI 问答和自动生成只能使用当前账号有权访问的数据。
- 不同资方之间默认数据隔离，除非管理层或管理员获得明确跨资方授权。
- 飞书群聊和文档应支持“指定空间 / 指定文件夹 / 指定群聊 / 指定 Base”粒度授权，避免把无关内部信息误作为上下文。
- 所有自动拉取、索引和生成行为应记录来源和时间，便于追溯。

## 4. 初步功能模块

### 4.1 战略合作知识库

- 沉淀合作历史沿革、会议纪要、项目资料、产品方案、客户进展、关键决策和风险事项。
- 支持按项目、客户、主题、会议、时间线进行归档。
- 支持自动总结、问答检索、汇报材料生成。

### 4.2 项目组合管理

- 管理 eCAS、Project Dscale、Digital Onboarding、MFP、Affluent、GTM 活动、SCV 投资协同等合作线。
- 每条合作线应有项目 owner、阶段状态、里程碑、下一步动作、风险等级和完成标准。

### 4.3 任务与会议 Follow-up

- 将会议纪要自动拆分为待办事项。
- 记录任务负责人、截止时间、依赖方、当前状态和逾期预警。
- 支持跨部门提醒和管理层查看。

### 4.4 客户与商机管理

- 对接 CRM 客户池，识别适合 SCB 合作产品的客户。
- 管理客户进件、开户、授信、放款、财富管理转化等状态。
- 支持客户标签、准入标准、客户优先级和转化漏斗。

### 4.5 业务数据仪表盘

- 展示项目进度、客户转化、进件数量、开户数量、授信金额、放款金额、资产池规模、逾期风险等指标。
- 支持管理层快速查看 SCB 合作整体健康度。

### 4.6 风险与卡点管理

- 跟踪 SCB 内部审批、技术联调、合规文件、客户材料、产品定价、系统对接等卡点。
- 支持风险分级、责任人、解决路径、预计完成时间和升级机制。

### 4.7 物料库

- 物料库用于管理活动、产品、客户沟通、内部操作和管理汇报相关材料。
- 物料生成是物料库中的一个子功能标签页，用于在活动或产品更新时，自动或半自动生成活动海报、产品 one-pager、内部操作指南、客户沟通材料、管理层汇报材料等。
- 生成后定稿的物料应进入物料库，按类型、关联项目 / 活动、适用对象、状态和版本号进行分类整理归档。
- 物料库应保留版本记录，包括版本号、修改人、修改时间、修改说明、审核状态和定稿状态。

## 4.8 MVP 模块优先级

第一版 MVP 建议包含 7 个核心模块：

| 优先级 | 模块 | 主要价值 |
| --- | --- | --- |
| 1 | SCB 总览仪表盘 | 查看目标、资产池、收入、开户、风险、里程碑和合作健康度 |
| 2 | 项目组合看板 | 管理 eCAS、Dscale、开户 / Digital Onboarding、MFP 等业务项目线 |
| 3 | 客户推进表 | 管理 Dscale 客户、开户客户、POC 客户、重点客户 LTV 和转化路径 |
| 4 | 任务与会议 Follow-up | 将会议纪要拆解为待办事项，跟踪负责人、截止时间、状态和逾期提醒 |
| 5 | 知识库 / AI 问答 | 沉淀会议、文档、产品方案、客户进展、关键决策，并支持自动总结与汇报 |
| 6 | 项目管理档案库 | 分类归档背景信息、协议文档、会议纪要、日常工作记录、产品资料和客户资料 |
| 7 | 市场活动及品牌共建 | 管理联合活动、品牌露出、客户邀约、产品发布、活动物料库和活动复盘 |

## 4.8.1 多资方复用逻辑

系统底座应按“通用框架 + 资方知识库 / 数据库 + 智能项目线生成”的方式设计。

通用框架：
- PMO 工作台。
- 管理层仪表盘。
- 项目组合看板。
- 客户推进表。
- 任务与会议 Follow-up。
- 知识库 / AI 问答。
- 项目管理档案库。
- 市场活动及品牌共建。

资方数据层：
- 每个战略合作资方拥有独立的知识库、项目资料、客户数据、任务数据、会议纪要和业务指标。
- 不同资方之间需要支持数据隔离。
- Dowsure 内部管理层可按权限查看单一资方或跨资方汇总视图。

智能生成层：
- 系统读取某个资方的知识库 / 数据库后，应能识别并生成适配该资方的项目线。
- 例如对接汇丰银行知识库后，系统可根据实际业务合作情况，生成信贷产品合作类、开户类、业务线基建类、市场活动合作类等项目管理模块。
- 管理层仪表盘保留基础框架，但具体指标、图表、日报 / 周报 / 月报、规划文档和风险提示，应根据该资方实际合作情况智能调整。

项目线类型示例：

| 项目线类型 | 示例 |
| --- | --- |
| 信贷产品合作类 | Dscale、大额信贷产品、Fastpay、其他融资产品 |
| 开户类 | 企业开户、企业主 / 高管开户、Digital Onboarding |
| 业务线基建类 | eCAS、Dowalet 账户体系、API 对接、资金路径建设 |
| 市场活动合作类 | GBA Summit、产品发布会、客户沙龙、联合 PR |
| 资本 / 战略背景类 | SCV 投资、集团级合作备忘录、战略 sponsor；默认作为背景或里程碑，归入项目管理档案库，不作为日常项目线 |
| 风险 / 合规协同类 | 合规审批、牌照路径、风控规则、责任边界 |

### 4.8.2 市场活动及品牌共建下的物料库

物料库不作为左侧一级菜单，而是作为“市场活动及品牌共建”模块下的子菜单 / 子标签页。

第一版物料库建议优先支持以下材料类型：

- 活动海报：例如 SCB GBA Summit、Project Dscale 官宣、客户邀约活动。
- 产品 one-pager：例如 Dscale 产品介绍、eCAS 账户方案、开户合作说明。
- 内部操作指南：例如产品进件操作指南、客户材料清单、RM / 销售协作 SOP。
- 客户沟通材料：例如客户邀约话术、开户材料说明、产品卖点摘要。
- 管理层汇报材料：例如周报、月报、项目进展摘要、关键风险和下一步行动。

第一版“市场活动及品牌共建”模块建议包含以下子菜单：

| 子菜单 | 功能 |
| --- | --- |
| 活动与品牌共建 | 统一管理活动事项、邀约名单、活动节点、执行任务、联合品牌露出、PR 口径、对外背书、媒体与活动资源 |
| 物料库 | 管理活动海报、one-pager、邀约话术、内部指南等物料 |
| 复盘记录 | 沉淀活动结果、客户转化、品牌露出和后续跟进任务 |

物料库内部建议包含以下标签页：

| 标签页 | 功能 |
| --- | --- |
| 物料归档 | 查看已定稿、可复用、草稿、待审核物料 |
| 物料生成 | 基于知识库、项目资料、客户画像、产品参数和活动安排生成新物料 |
| 审核与定稿 | 跟踪品牌、合规、业务 owner 的审核状态 |
| 版本记录 | 记录版本号、修改人、修改时间、修改说明和定稿状态 |

物料生成子功能应尽量复用知识库中的项目资料、客户画像、产品参数、活动安排和历史版本，避免每次从零开始写材料。

子菜单交互要求：
- “市场活动及品牌共建”下的活动与品牌共建、物料库、复盘记录应是真实可切换的子标签页。
- 点击“物料库”子标签页后，应展示物料归档、物料生成、审核定稿、版本记录等完整内容，而不是只保留入口按钮。
- 每个按钮、标签、卡片、勾选框点击后都应有明确反馈，例如 toast、状态变化、打开详情、打开 AI 协作、切换标签或创建草稿。

## 4.8.3 SCB 第一版项目组合看板

SCB 作为第一阶段样板资方，项目组合看板建议先包含以下业务项目线：

| 项目线 | 类型 | 管理说明 |
| --- | --- | --- |
| eCAS 账户体系 | 业务线基建类 | Dowalet 底层账户基础设施，支撑收还款、结算、后续放款及资金路径 |
| Project Dscale | 信贷产品合作类 | 大额跨境多币种信贷产品，目标推动产品分层、标准化、数字化及资产池规模增长 |
| 企业开户 / Digital Onboarding | 开户类 | 企业开户 POC、标准 SOP、线上化进件及 API 数据互通 |
| 企业主 / 高管个人开户 | 开户类 | 包含 Affluent 财富管理相关转化场景，但不单独作为当前核心项目线 |
| MFP 长期合作 | 信贷 / 结算长期合作类 | 当前处于合作提案阶段，作为长期目标管理，不作为短期执行主线 |

SCV 投资协同不进入 SCB 日常项目组合看板，应作为战略背景信息、重大里程碑和资本资源投入观察项归档到项目管理档案库。

Referral Agreement 等协议规则也不作为项目组合看板中的子项目展示，应归入项目管理档案库，并作为知识库问答、汇报生成和收入规则解释的底层资料。

市场活动及品牌共建不放在项目组合看板中作为普通项目线管理，而是作为系统左侧一级工作模块，与项目组合、客户推进、知识库、物料库等模块并列。0714 SCB GBA Summit & Project Dscale PR 是该模块下的一次具体市场活动事项。

### 4.8.4 SCB 项目线管理深度

第一版系统中，SCB 项目线应分为“深度管理”和“轻量管理”两层。

深度管理项目线：

| 项目线 | 深度管理原因 |
| --- | --- |
| eCAS 账户体系 | 是 Dowalet 业务线基础设施，影响后续收还款、结算、放款及资金路径 |
| Project Dscale | 是 2026 年核心规模化目标，直接关联资产池规模和放款收入 |

轻量管理项目线：

| 项目线 | 轻量管理原因 |
| --- | --- |
| 企业开户 / Digital Onboarding | 当前仍处于 POC 与方案探索阶段，先轻量追踪开户进度、SOP 和后续线上化节点 |
| 企业主 / 高管个人开户 | 当前归入开户延伸场景，暂不作为独立重运营主线 |
| MFP 长期合作 | 当前仍处于合作提案阶段，作为长期目标和背景机会管理 |

独立深度工作模块：

| 模块 | 深度管理原因 |
| --- | --- |
| 市场活动及品牌共建 | 承担客户邀约、产品官宣、品牌背书和 GTM 转化，0714 GBA Summit 是重要节点；需要独立管理活动、物料、邀约、传播和复盘 |

## 4.8.5 项目管理档案库

项目管理档案库用于统一归档 SCB 合作相关的背景资料、协议文档、会议纪要和日常工作记录。它不是项目组合看板，也不直接代表业务推进状态，而是知识库和 AI 能力的底层资料来源。

第一版档案分类建议：

| 分类 | 示例 |
| --- | --- |
| 背景信息 | SCV 投资协同、合作历史沿革、战略资源投入背景 |
| 协议文档 | MOU、Referral Agreement、开户补充协议、合作条款 |
| 会议纪要 | 0424 SCB 高层 Workshop、周会纪要、产品会、客户会 |
| 日常工作记录 | 飞书群关键沟通、邮件往来、卡点记录、决策记录 |
| 产品资料 | Dscale 产品手册、eCAS 方案、MFP 合作提案 |
| 客户资料 | POC 客户材料、客户画像、准入标准、推进记录 |

档案库页面层级：
- 第一层应先展示档案大类，例如战略背景、协议、纪要、提案、产品资料、客户资料。
- 用户点击某个分类标签或分类卡片后，进入该分类详情页。
- 分类详情页中再展示具体文档标题，例如 Referral Agreement、0424 SCB 高层 Workshop、MFP 产品合作方案等。
- 具体文档详情页展示正文、摘要、版本、来源、关联对象、AI 总结和可拆解任务。
- 档案库主界面不应把归档规则作为主要内容展示；归档规则更适合作为后台规则、帮助说明或设置项。
- 主界面右侧优先展示可操作内容，例如待处理档案、最近更新、待分类资料、AI 摘要入口和批量归档入口。

档案库中的每份资料应支持关联对象：
- 项目线。
- 客户。
- 活动。
- 任务。
- 物料。
- 风险事项。

档案库应服务于以下能力：
- 知识库问答。
- 自动总结。
- 会议纪要拆任务。
- 周报 / 月报 / 汇报材料生成。
- 项目复盘。
- 决策依据追溯。

## 4.9 首页设计原则

第一版系统默认首页采用“PMO 工作台”，服务日常项目推进；同时保留“管理层仪表盘”作为汇报视图。

### 4.9.0 视觉主题

系统应支持深浅两个视觉版本：

| 主题 | 参考来源 | 设计方向 |
| --- | --- | --- |
| 深色版 | 0424 SCB workshop PPT 封面 | 深海军蓝底，白色高对比标题，青绿、蓝色、玫红作为流线感强调色，适合大屏展示、活动场景和沉浸式汇报 |
| 浅色版 | 0424 SCB workshop PPT 内容页 | 白底、细边框、克制商务信息密度，适合 PMO 日常工作台、表格、任务和项目管理 |

主题切换不应只是简单反色，而应保留两套场景化视觉风格：
- 浅色版优先保证信息扫描效率。
- 深色版优先体现 Dowsure + SCB 联合品牌感和高层汇报质感。
- 两套主题的模块结构、字段和交互保持一致。

### 4.9.1 默认首页：PMO 工作台

PMO 工作台应优先回答：

- 今天我必须推进什么？
- 哪些任务已经逾期或即将逾期？
- 哪些项目出现卡点？
- 哪些客户需要跟进？
- 哪些会议事项尚未闭环？
- 哪些材料需要生成或更新？

建议首页组件：

| 组件 | 内容 |
| --- | --- |
| 今日待办 | 今日需要完成或推动的任务 |
| 逾期 / 风险事项 | 逾期任务、卡点、需升级事项 |
| 项目卡点 | eCAS、Dscale、开户、GBA Summit 等项目的当前阻塞点 |
| 客户推进提醒 | Dscale 客户、开户客户、POC 客户的下一步动作 |
| 会议 Follow-up | 最近会议拆解出的待办和未闭环事项 |
| 待生成材料 | 活动海报、one-pager、操作指南、汇报材料等生成入口 |
| 快速问答 | 直接向 SCB 知识库提问或生成摘要 |

导航交互原则：
- 左侧一级菜单应固定在视口内，不随右侧内容上下滚动。
- 右侧主内容区域独立滚动，保证用户在长页面中随时可以切换模块。
- 移动端可折叠左侧菜单，桌面端默认常驻。

### 4.9.2 汇报视图：管理层仪表盘

管理层仪表盘应优先回答：

- SCB 合作整体是否健康？
- 2026 年目标推进到哪里？
- 资产池、收入、开户、关键客户是否达标？
- 哪些事项需要管理层介入？
- SCB 投入的战略资源是否符合预期？

管理层视图不作为默认首页，但应支持快速切换，并能够自动生成周报、月报或专项汇报。

汇报视图不应是静态按钮或单纯切换排版，而应作为“引导式汇报生成器”展开：

- 先引导用户选择汇报对象，例如 Dowsure 管理层、Byron / CEO、SCB 高层或内部跨部门。
- 再确认汇报类型和周期，例如日报、周报、月报、专项汇报或会议简报。
- 再选择本次重点，例如 eCAS 上线卡点、Dscale 资产池、0714 活动筹备、客户推进或战略资源投入。
- 如果关键数据缺失，AI 应先追问或提示需要补齐的数据，而不是直接生成空泛汇报。
- 生成结果应包含核心指标、关键进展、风险 / 需升级事项、管理层决策点和下一步行动。

## 4.10 任务来源与演进路径

### 4.10.1 产品原则

任务系统的建设原则是：先实现自动化，再实现智能化。

第一版 MVP 应先保留完整的信息架构、按钮和模块入口，但功能实现从最简单、最可靠的手动添加任务开始，后续逐步扩展自动拆解、外部同步和智能建议能力。

### 4.10.2 任务来源

系统最终应支持以下任务来源：

| 来源 | 说明 | MVP 阶段 |
| --- | --- | --- |
| 手动添加任务 | 用户在 PMO 工作台、项目页、客户页或对话悬窗中手动创建任务 | MVP 1.0 必须实现 |
| 会议纪要拆解 | 根据会议纪要自动识别 action items、负责人、截止时间和依赖事项 | MVP 1.1 |
| 文档拆解 | 根据产品方案、活动方案、项目文档拆解任务 | MVP 1.1 / 1.2 |
| 项目里程碑倒推 | 根据 eCAS 上线、GBA Summit、Dscale 放款等里程碑倒推关键任务 | MVP 1.2 |
| 客户推进表下一步动作 | 从客户推进表中的下一步动作自动生成或提醒任务 | MVP 1.2 |
| 飞书任务同步 | 与飞书任务打通，读取或同步任务状态 | MVP 1.3 |
| 飞书群消息同步 | 从指定飞书群消息中识别任务、承诺、卡点和待跟进事项 | MVP 1.3 / 1.4 |
| 系统风险提醒 | 自动发现项目或客户风险，例如长期未更新、临近 deadline、SCB 超时未回复等 | MVP 1.4 |
| 智能任务建议 | 根据风险、上下文和历史推进经验，建议应发起的任务和 owner | MVP 1.5 |

### 4.10.3 对话悬窗设计

任务系统应设计一个类似对话悬窗的交互入口：

- 默认收起在页面右侧边框。
- 鼠标经过或点击后展开。
- 可在任意页面快速创建任务、提问、生成材料、总结当前页面、拆解会议纪要。
- 展开后应支持自然语言输入，例如“帮我给 eCAS 技术联调创建一个下周五前完成的任务，owner 是 Kevin”。
- 创建任务时，应允许选择关联对象：项目、客户、会议、文档、材料或风险事项。
- 第一版先实现手动添加任务和关联对象选择；后续再接入自动拆解和智能建议。
- AI 协作悬窗应根据当前页面内容定制化调整，不同页面显示不同的提示文字、快捷按钮、默认任务类型和输入示例。
- 例如在“市场活动及品牌共建”页面，AI 应优先提示缺失物料、活动海报、one-pager、客户邀约名单、品牌口径和复盘；在“项目管理档案库”页面，AI 应展示归档规则、分类建议、摘要生成、关联对象建议和任务拆解。

交互形态采用“混合型”：

- 上方为常用快捷按钮。
- 下方为自然语言对话输入框。
- 快捷按钮用于高频确定性操作；自然语言输入用于复杂任务、总结、生成和拆解。

第一版建议快捷按钮：

| 按钮 | 功能 |
| --- | --- |
| 新建任务 | 手动创建任务，选择 owner、截止时间、关联对象 |
| 上传纪要 | 上传或粘贴会议纪要，后续用于拆解任务 |
| 生成材料 | 进入活动海报、one-pager、操作指南、汇报材料生成流程 |
| 问知识库 | 向 SCB 知识库提问 |
| 写周报 | 基于项目、客户、任务和风险生成周报草稿 |

自然语言输入示例：

- “帮我给 eCAS 技术联调创建一个下周五前完成的任务，owner 是 Kevin。”
- “总结一下 Project Dscale 目前最大的三个卡点。”
- “把这段会议纪要拆成待办事项。”
- “基于 GBA Summit 信息生成一版客户邀约海报文案。”
- “列出过去 7 天没有更新的 Dscale 客户。”

### 4.10.4 页面级 AI 协作上下文

每个一级模块的 AI 协作悬窗应有独立上下文配置：

| 页面 | AI 协作重点 |
| --- | --- |
| PMO 工作台 | 今日待办、逾期事项、项目卡点、日报 / 周报摘要 |
| 项目组合 | 项目健康度、里程碑倒推、owner 检查、项目风险 |
| 客户推进 | 客户下一步动作、材料缺口、客户分层、跟进话术 |
| 任务 Follow-up | 新建任务、会议纪要拆解、逾期检查、owner 建议 |
| 知识库问答 | 知识问答、事实抽取、文档总结、知识缺口识别 |
| 项目管理档案库 | 归档规则说明、分类建议、摘要生成、关联对象建议、决策提取 |
| 市场活动及品牌共建 | 缺失物料检查、活动海报、one-pager、邀约名单、品牌口径、活动复盘 |

档案库页面的归档规则不应占据主界面，而应作为 AI 协作展开后的基础说明或帮助信息出现。

第一版建议产品线：

用户点击某个产品线后，再进入对应客户跟进表详情。客户跟进表字段可从飞书多维表格数据源同步，当前参考数据源：
